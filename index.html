Proof of Concept — Internal Review

Kendo UI Feasibility Assessment
AuditComply Template Builder

Can Kendo UI for React replicate the AuditComply template builder interface to a production standard? This report presents the evidence.

Finding

Kendo is not a practical fit for this interface

A structured proof-of-concept was built across four progressive versions, each pushing Kendo's customisation further. The conclusion is consistent at every level.

Kendo UI components render fundamentally different HTML to what the AuditComply design system was built to target. The portal's own compiled CSS — the ground truth for how this interface should look — cannot reach Kendo's internal elements without 16 !important overrides on Kendo's internal class names. At the point where a like-for-like result is achieved, Kendo accounts for just 20% of the implementation, all of it carrying ongoing upgrade risk. The existing AuditComply design system already contains everything needed to style this interface correctly.

The Target

What the interface must look like

Recreated from the portal's compiled CSS (template-builder1_5.css) and the actual Django template HTML. Every measurement, colour and icon is sourced directly from the portal's SCSS files. This is the benchmark each version is compared against.

Target: section header with 120px stripe, requirement cards with attribute rows, Select2 dropdowns, build-toggle switches, and mandatory columns.

V3 Best-Effort Kendo — Natural Stopping Point

Brand tokens applied, section header approximated via ExpansionPanel's title prop, modest CSS overrides for the most obvious mismatches. This is what a competent developer would produce after 2–3 days of genuine Kendo customisation work.

This is where a developer would stop and raise a concern. The visual output is recognisably Kendo, not recognisably the portal. Closing the remaining gap — the 120px section stripe, the 10px toggle track, the Select2 layout, the 55×55px icon buttons — requires moving from "customising Kendo" to "writing custom code that wraps Kendo."
V4 Like-for-Like — Maximum Kendo Result

The portal's own compiled CSS is loaded alongside Kendo. Kendo components handle interactive behaviour (drag-and-drop, toggle state, dropdown state). The portal CSS handles all structural styling. A separate kendo-overrides.css file compensates for where Kendo's HTML conflicts with the portal's CSS selectors.

Kendo usage25%
Custom CSS~130 lines overrides
!important on Kendo16
Visual match~90%
Cumulative effort11 days
The decisive finding: To reach ~90% visual match, 80% of the work is done by the portal's own CSS — not Kendo. The 6 remaining Kendo components (Sortable, Switch, Checkbox, DropDownList, MultiSelect, Button) still require 16 !important overrides on Kendo's internal class names because Kendo's HTML does not match what the portal CSS targets. Every Kendo minor release risks silently breaking these overrides. Two further planned Kendo components were replaced with custom implementations during development.

Recommendation

Conclusion

Based on the evidence across all four versions.

Kendo UI is not a practical fit for the template builder interface

Kendo UI for React provides real value for data-heavy, grid-centric interfaces where its default component structure aligns with the target design — data tables, charts, schedulers. For an interface with an established custom design system like the AuditComply template builder, it works against the existing design system.

The portal's compiled CSS is the ground truth for how this interface must look. Any approach that renders HTML matching its selectors requires zero additional overrides. Kendo cannot replicate the portal's design without overriding its own internal component structure — its components output .k-switch-track, .k-dropdownlist, and .k-item wrappers that require 12 !important overrides tied to Kendo's internal class names, each one a silent failure risk on Kendo upgrades.

At the like-for-like stage, Kendo accounts for 20% of the implementation and introduces commercial licensing costs, a cascade of undocumented peer dependencies, and ongoing maintenance risk. The 80% doing the real work is the portal's own CSS.
